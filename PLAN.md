# RF·MESH 无线电监测网格系统 — 工程任务分解

> 文档版本：v1.0 | 日期：2026-03-13

---

## 一、系统定位与核心架构

### 整体拓扑

```
[ EM550接收机 ]
      │ SCPI / TCP
[ 边缘节点软件 (Edge Agent) ]   ← 部署在每个监测站点的服务器
      │ REST + WebSocket（上行数据 + 心跳 + 实时流）
      ↕
[ 云端服务器 (Cloud Server) ]   ← 局域网/专网内的中心服务器
      │ WebSocket
[ 云端前端 (Web UI) ]
```

### 核心设计原则

- **边缘自治**：Edge Agent 在无云端指令时独立执行全频段常规扫描
- **云端协调**：Cloud Server 负责任务下发、数据汇聚、统计展示
- **数据精简上传**：边缘做预处理 + 聚合，每分钟上传压缩统计包，不传原始 I/Q
- **实时流按需触发**：实时频谱流仅在用户主动请求时开启，不常驻占用带宽

## 二、组件划分

### 2.1 边缘节点软件（Edge Agent）

| 子模块 | 职责 |
|--------|------|
| SCPI 驱动层 | 封装 EM550 SCPI 指令；TCP 连接管理；指令队列 |
| 扫描调度器 | 任务队列管理；常规扫描与云端任务的优先级切换 |
| 全频段扫描引擎 | 20MHz–8GHz，25KHz 步进，逐段扫描，输出原始频谱帧 |
| 频谱预处理器 | 按频段划分规则合并 25K 步进；运营商频段聚合处理 |
| 1分钟聚合器 | 滑动窗口最大值统计；打包为上传数据包 |
| 云端通信模块 | 上行：REST 上传聚合包；下行：WebSocket 接收任务指令 |
| 实时流模块 | 按任务参数对指定频段做连续扫描，WebSocket 推送频谱帧到云端 |

### 2.2 云端后端（Cloud Backend）

| 子模块 | 职责 |
|--------|------|
| 站点管理 | 站点注册、心跳、在线状态维护 |
| 数据接收 API | 接收边缘上传的聚合包；写入时序数据库 |
| 数据查询 API | 按站点/频段/时间段查询历史统计数据 |
| 任务调度系统 | 生成任务、下发到目标站点、追踪执行状态 |
| 频率指配模块 | 两频段输入 → 逐信道空闲度计算 → 结构化结果输出 |
| 批量站点扫描模块 | 中心频点+跨距+分析带宽 → 批量下发 → 汇聚各站最大值 |
| 实时频谱代理 | 转发边缘实时流到前端 WebSocket |
| WebSocket 网关 | 前端连接管理；推送任务状态、实时频谱、在线状态变更 |

### 2.3 云端前端（Web UI）

| 页面/组件 | 职责 |
|-----------|------|

## 三、关键数据流详解

### 3.1 常规扫描数据流（持续运行）

```
EM550
  → [SCPI] → 边缘扫描引擎（25KHz步进全频段扫描）
  → 预处理器（频段合并 + 运营商段聚合）
  → 1分钟聚合器（取最大值，生成聚合包）
  → REST POST /api/stations/{id}/spectrum-snapshot
  → 云端写入时序数据库
```

**聚合包数据结构（每分钟一包）：**
```json
{
  "station_id": "ST-01",
  "timestamp": "2026-03-13T08:00:00Z",
  "duration_sec": 60,
  "entries": [
    { "freq_mhz": 87.5,  "bw_khz": 200, "max_dbm": -72.3 },
    { "freq_mhz": 97.4,  "bw_khz": 200, "max_dbm": -41.1 },
    ...
  ]
}
```

### 3.2 频率指配流程

```
用户输入：
  上行频段：935–960 MHz
  下行频段：890–915 MHz
  信道带宽：200 KHz

云端计算：
  → 按信道带宽切分两个频段
  → 查询各信道最近N分钟统计数据（所有站点）
  → 输出对照表：信道编号 | 上行最大电平 | 下行最大电平 | 空闲判断

前端展示：
  上行 | Ch1 -105dBm ✓  Ch2 -98dBm ✓  Ch3 -41dBm ✗ ...
  下行 | Ch1 -103dBm ✓  Ch2 -97dBm ✓  Ch3 -39dBm ✗ ...
```

### 3.3 批量站点扫描任务流程

```
用户输入：
  中心频率：433.92 MHz
  跨距：±5 MHz（即 428.92–438.92 MHz）
  分析带宽：25 KHz
  时长：5 分钟
  参与站点：全选 / 手动选

云端 → 下发任务到所有选中站点（WebSocket）
边缘 → 在指定跨距内执行扫描任务（优先级高于常规扫描）
边缘 → 统计分析带宽内最大值 → 上报到云端

云端展示：
  站点列表 | 最大电平 | 扫描完成时间 | 实时回放按钮

用户点击「实时回放」：
  → 云端向该站点下发实时流任务
  → 边缘启动连续扫描并推送频谱帧
  → 前端显示滚动瀑布图/频谱图
```

## 四、数据压缩策略

> **架构决策（已确认）：边缘端只做时间维度压缩，不做频率维度合并。**
> 信道划分、频段聚合等频率维度操作全部在云端按需执行，理由：
> - 频段规划参数随时可能调整；云端统一管理，改一处全网生效
> - 边缘端按固定 25 kHz 粒度上传，云端存储原始数据，历史数据可用任意新规则重算

### 4.1 模式一：闲时扫描（Background Scan，常驻后台）

Edge 无云端任务时持续执行全频段扫描，并对扫描结果做**时间维度压缩**后上传：

| 参数 | 值 | 说明 |
|------|----|------|
| 扫描范围 | 设备全频段（EM550: 20MHz–3.6GHz） | 固定，不可动态调整 |
| 步进 | 25 kHz | 固定，上传原始粒度 |
| 时间窗口 | 60 秒（可配置） | 窗口内多次扫描结果聚合为一帧 |
| 聚合方式 | 每 bin 取最大值（max-hold） | 保证不遗漏任何出现过的信号 |
| 上传内容 | 全频段 25 kHz 粒度"1分钟最大值快照" | gzip 压缩后约 100–200 KB/帧 |

### 4.2 模式二：任务模式（Task Mode，云端下发触发）

Cloud 向 Edge 下发任务后，Edge 暂停闲时扫描，执行任务，并以**实时流**方式将结果发往云端：

| 参数 | 说明 |
|------|------|
| 触发方式 | 云端通过 WebSocket 下发 `task` 消息 |
| 传输方式 | 实时流（不走每分钟聚合上传） |
| FPS | 可配置：1 / 10 / 30 fps，或最大速率；**默认 10 fps** |
| 聚合窗口 | = 1/FPS 秒（如 10fps 则每 100ms 聚合一次） |
| 聚合方式 | max-hold（默认）或 average（可配置） |
| 目的 | FPS 参数同时控制带宽占用和展示精度 |

### Phase 1：边缘扫描引擎（约 2 周）

> 目标：Edge Agent 可驱动 EM550 完成全频段扫描并上传聚合数据

| # | 任务 | 说明 |
|---|------|------|
| 1.1 | SCPI 驱动封装 | TCP 连接、重连、指令发送/响应解析；封装为 `BaseSpectrumDriver` 抽象类 |
| 1.2 | EM550 驱动实现 | PSCan / IFPAN / FSCan 三种模式；dBµV→dBm 转换；自动分段拼接 |
| 1.3 | RSA306B 驱动实现 | USB + tekrsa-api-wrap；40MHz 分段拼接；band_scan / if_analysis / channel_scan |
| 1.4 | 全频段扫描循环 | 驱动层分段拼接完整频谱，上层无感；任务优先级高于后台扫描 |
| 1.5 | 1分钟聚合器 | 滑动时间窗口；每 bin 取最大值；生成聚合包（不做频率合并） |
| 1.6 | 任务模式实时流 | 按 FPS（默认10fps）聚合发送；max-hold / average 可选 |
| 1.7 | 聚合包上传 | REST POST；失败重试（指数退避）；本地缓存（断网补传） |
| 1.8 | 模拟器（Mock EM550） | 用于无硬件环境下的开发测试；返回随机频谱数据 |

**关键参数：**
- 20MHz–8GHz，步进 25KHz → 约 **319,200** 个原始数据点
- EM550 单次扫描跨距上限（待查文档确认），需分段扫描
- 估算单次全频段扫描时长：取决于接收机扫描速度，需实测

## 七、技术栈选型

| 层级 | 选型 | 理由 |
|------|------|------|
| Edge Agent | Python 3.11 + asyncio | 异步 I/O 适合 SCPI + WebSocket 并发；生态丰富 |
| Cloud Backend | FastAPI + Python 3.11 | 自动 API 文档；WebSocket 支持好；类型安全 |
| 时序数据库 | TimescaleDB（PostgreSQL 扩展） | 时序查询性能；SQL 接口方便查询；降采样内置 |
| 消息/任务队列 | 内置 asyncio Queue（初期）| 初期规模不需要 Redis/Kafka；后期可替换 |
| 前端框架 | Vue3 + Vite + TypeScript | 生态成熟；Composition API 利于状态管理 |
| 图表 | ECharts | 频谱/瀑布图支持好；Canvas 渲染性能优 |
| 部署 | Docker Compose | 一键启动；各服务隔离 |

| 4 | 云端服务器与各边缘站点的网络拓扑 | 影响 WebSocket 方向（Edge→Cloud 还是 Cloud→Edge 发起） |
| 5 | 频段合并规则的完整业务需求 | band_rules.yaml 的内容 |
| 6 | 实时流的帧率要求（几帧/秒） | 影响带宽和前端渲染策略 |
| 7 | 数据保留周期要求 | 影响存储容量规划 |

## 九、开发顺序建议

```
Phase 0（框架） → Phase 1（边缘扫描）+ Phase 2（云端存储）[并行]
                                    ↓
                     Phase 3（前端基础）+ Phase 4（频率指配）[并行]
                                    ↓
                               Phase 5（批量扫描 + 实时流）
                                    ↓
                               Phase 6（工程化，持续穿插）
```

Phase 1 和 Phase 2 可以两人并行开发，Phase 3/4 可以前端工程师在 Phase 2 完成后同步展开。
