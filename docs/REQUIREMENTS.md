# RF·MESH 无线电监测网格系统 — 需求与设计说明

> 文档版本：v0.3 | 最后更新：2026-03-13
> 性质：研发对齐文档，记录架构决策与功能需求，随沟通持续更新

---

## 1. 系统定位

RF·MESH 是一套部署于专网环境的多站点无线电监测平台，由边缘采集节点与中心云端服务器协同构成。系统的核心目标是：

- **持续积累**：各站点不间断扫描，形成全频段历史电平数据库
- **异常发现**：通过模板匹配与 AI 辅助，自动识别频谱中的异常信号
- **任务驱动**：支持对特定频段发起定向扫描任务，聚合多站结果
- **台站建库**：逐步建立覆盖辖区内所有已知信号的台站信号库

---

## 2. 整体架构

### 2.1 拓扑结构

```
[ 监测设备（各厂商） ]
        │ SCPI / 私有协议
[ 边缘节点软件 (Edge Agent) ]    ← 每个监测站点部署一套
        │ 向上主动建立连接（WebSocket + REST）
        ↓
[ 云端服务器 (Cloud Server) ]    ← 专网内中心服务器
        │ WebSocket
[ 云端前端 (Web UI) ]
```

**连接方向约定：** Edge 主动向 Cloud 注册并建立长连接，Cloud 通过已建立的连接向 Edge 下发任务。这一设计简化了多站点部署，避免 Cloud 需要逐一主动连接各站点内网 IP。

**网络规划：** 通过隧道技术将各站点与云端服务器纳入同一扁平化局域网（不同网段），降低穿透复杂度。

### 2.2 设备驱动抽象层

系统需兼容多厂商监测设备，因此边缘软件设计**驱动抽象层**：

- 定义统一的设备驱动接口（如 `scan_range(start, stop, rbw)` → 返回标准格式频谱帧）
- 初期适配一款易于操作的设备作为参考实现
- 后续新设备仅需实现同一接口，通过配置文件切换驱动，不影响上层逻辑
- 各设备的特有限制（如单次扫描最大跨距、IF 分析带宽）由驱动层内部处理，上层无感

## 3. 边缘节点软件（Edge Agent）

### 3.1 常规全频段扫描

边缘节点在无任务下发时，持续执行全频段（20 MHz – 8 GHz）背景扫描。

**扫描参数：**
- 原始步进：25 kHz（全频段共约 319,200 个数据点）
- 驱动层按设备能力自动分段，上层统一请求全频段

**预处理（频段合并）：**
原始 25 kHz 步进不直接上传，按频段业务规划合并为信道粒度数据。合并规则以配置文件定义，可热更新。典型规则：

| 频段 | 业务 | 合并粒度 |
|------|------|----------|
| 87.5–108 MHz | FM 广播 | 200 kHz（8×25K） |
| 118–137 MHz | 航空 VHF | 25 kHz（保持原始精度） |
| 350–470 MHz | 对讲/PDT | 12.5 kHz / 25 kHz |
| 运营商 4G/5G 频段 | 移动通信 | 整段聚合（不按信道） |
| 其余兜底 | — | 100 kHz（4×25K） |

**1 分钟聚合上传：**
- 时间窗口：1 分钟
- 聚合方式：每个信道取窗口内最大电平值
- 上传内容：结构化聚合包（见 §7.1），体积远小于原始数据

### 3.2 任务执行模式

Cloud 下发扫描任务时，Edge 暂停常规扫描，执行任务扫描，完成后恢复常规扫描。任务参数包括：中心频率、跨距、分析带宽、扫描时长。

### 3.3 实时频谱流

当 Cloud 下发实时流任务时，Edge 对指定频段做连续扫描，并将频谱帧实时推送至 Cloud。

**帧率控制：**
- 设备本身以固有速度产出数据，Edge 软件在发送前进行节流
- 可配置目标帧率：
  - `0`：不节流，按设备最快速度发送
  - 默认值：5 fps
  - 推荐范围：5–10 fps（可在前端或任务参数中调整）
- 后台统计任务（非实时查看）：1 帧/分钟

## 4. 云端功能模块

### 4.1 站点总览仪表盘

- 展示所有已注册站点的在线状态、最后心跳时间、今日数据点数等
- 实时推送站点状态变更

### 4.2 频率指配工具

**场景：** 需要为某业务分配工作频率，查询现有频段内哪些信道当前空闲。

**输入参数：**
- 上行频段范围（起止频率）
- 下行频段范围（起止频率）
- 信道带宽
- 查询时间窗（最近 N 分钟历史数据）
- 空闲判定阈值（默认 -95 dBm，可调）
- 参与统计的站点（默认全选）

**输出：**
- 信道对照表：上下行对称展示，每个信道显示最大电平值及空闲/占用判断
- 可导出 CSV，供出具频率分配报告使用

### 4.3 批量站点扫描任务

**场景：** 对特定频段发起面向全部或指定站点的定向扫描，汇聚多站结果，判断信号的空间分布。

**输入参数：**
- 中心频率
- 跨距（扫描范围 = 中心频率 ± 跨距/2）
- 分析带宽（≤ 跨距，统计此带宽内的最大值）
- 扫描时长
- 参与站点（默认全选）

**执行流程：**
1. Cloud 将任务下发至各选中站点
2. 各 Edge 执行扫描，统计分析带宽内最大电平值，完成后上报结果
3. Cloud 汇聚展示：站点列表 + 各站最大电平对比

**实时回放：**
在汇总结果页，点击任意站点的「实时回放」按钮，触发该站点启动实时频谱流，前端显示滚动频谱图/瀑布图。回放参数沿用任务的频段设置，帧率可单独调整。

### 4.4 历史数据回放

**场景：** 回溯某频段在过去某段时间内的信号变化情况。

**输入参数：**
- 中心频率
- 分析带宽（可另加可选跨距，用于查看历史频谱图）
- 历史时间范围（起止时间）
- 优先展示站点（默认展示全部）

**展示层 1 — 电平时间线（折线图）：**
- 横轴：所选时间范围
- 纵轴：最大电平（dBm）
- 每个站点一条折线（可切换单站/多站叠加）
- 鼠标悬停显示精确时间与电平值
- 可快速识别信号何时出现、何时消失、强弱变化规律

**展示层 2 — 历史频谱快照（点击展开）：**
- 点击时间线上任意时刻 → 弹出该时刻的频谱图
- 展示以中心频率为中心、跨距范围内的完整频谱形态
- 用于判断信号特征：宽带/窄带、持续发射/跳频突发、单一/多源等
- 要求边缘在常规扫描时以较低频率存储全频谱快照（或至少存储触发条件下的快照）

## 5. AI 辅助信号分析

### 5.1 设计思路

直接用 AI 对全频段频谱图做端到端的信号识别（数量、位置、带宽）技术难度极高，工程上不可靠。采用**分层处理**方式：

```
原始频谱数据
    │
    ▼
[第一层：模板匹配初筛]
    按频段规划模板，逐信道判断是否有信号占用
    → 输出：信号列表（频率、估算带宽）
    │
    ▼
[第二层：异常检测]
    对比信道规划：
      - 带宽异常（如 25K 对讲频段出现 100K 宽信号）
      - 电平异常（超出历史基线）
      - 频率偏移异常（信号不在标准信道中心）
    → 输出：异常信号列表，提取频谱截图 + 瀑布图截图
    │
    ▼
[第三层：AI 初判]
    将异常信号的频谱图 + 瀑布图送入图像识别 AI
    结合频段规划知识库（可联网检索补充）
    → 输出：信号类型初步判断 + 置信度 + 依据说明
    │
    ▼
[第四层：人工核实]
    技术人员对 AI 初判结果进行核实，确认或修正
    → 确认后入库，更新台站信号库
```

### 5.2 台站信号库建设目标

台站信号库是系统长期积累的核心资产，分三类管理：

| 类别 | 说明 | 处理方式 |
|------|------|----------|
| AI 自动归档 | 置信度高、特征明确的已知信号 | AI 直接入库，人工抽查 |
| AI + 人工核实 | AI 有初判但置信度中等 | 推送待核实队列，人工确认后入库 |
| 人工处理 | 高异常程度或 AI 无法判断 | 推送处置工单，人工分析后手动录入 |

最终目标：辖区内每个出现过的信号都有对应记录，包含频率、带宽、调制方式、业务归属、历史出现规律等。

## 6. 数据存储设计

### 6.1 存储层次划分

| 数据类型 | 存储方式 | 保留策略 | 说明 |
|----------|----------|----------|------|
| 1 分钟聚合统计包 | TimescaleDB（时序主表） | 滚动保留 N 天（可配置） | 用于历史电平趋势查询、频率指配 |
| 全频谱快照 | 对象存储 / 文件系统（压缩） | 保留 24–72 小时（可配置） | 用于历史回放点击查看频谱图 |
| 触发快照（告警/任务） | 同上，但永久保留 | 与告警/任务记录关联 | 证据留存 |
| 台站信号库 | PostgreSQL（关系型） | 永久 | 核心业务数据 |
| 任务记录 | PostgreSQL | 永久 | 可追溯 |
| 告警记录 | PostgreSQL | 永久 | 可追溯 |

### 6.2 核心数据模型

**时序聚合表（spectrum_stats）：**
```
time          TIMESTAMPTZ   -- 分钟级时间戳
station_id    TEXT
freq_mhz      FLOAT         -- 信道中心频率
bw_khz        FLOAT         -- 信道带宽
max_dbm       FLOAT         -- 该分钟内最大电平
```

**全频谱快照（spectrum_snapshots）：**
```
snapshot_id   UUID
station_id    TEXT
captured_at   TIMESTAMPTZ
freq_start    FLOAT         -- 快照频率起点（MHz）
freq_end      FLOAT         -- 快照频率终点（MHz）
resolution    FLOAT         -- 实际分辨率（kHz）
data_path     TEXT          -- 压缩数据文件路径
```

**台站信号库（signals）：**
```
signal_id         TEXT (主键, SIG-XXXXXX)
freq_center_mhz   FLOAT
bandwidth_khz     FLOAT
modulation        TEXT
signal_type       TEXT         -- 业务类型
status            ENUM(auto_archived, pending_review, manual)
confidence        FLOAT        -- AI 置信度
first_seen        TIMESTAMPTZ
last_seen         TIMESTAMPTZ
occurrence_count  INT
ai_conclusion     TEXT
tags              TEXT[]
notes             TEXT
```

### 6.3 全频谱快照的存储策略

全频段快照数据量大（每站每帧约 1–2 MB 压缩后），不宜对每帧都持久化。建议：

- **常规模式**：以低频率（如每 5 分钟一次）存储一帧全频谱快照，用于历史回放点击查看
- **任务/告警触发模式**：任务执行期间或异常触发时，提高快照存储频率（如每分钟或每次扫描完成即存）
- **存储格式**：频率轴 + 电平数组，使用 NumPy `.npz` 或 MessagePack 压缩，避免 JSON 冗余

## 7. 关键接口协议

### 7.1 聚合包格式（Edge → Cloud，每分钟上传）

```json
{
  "station_id": "ST-01",
  "timestamp": "2026-03-13T08:00:00Z",
  "duration_sec": 60,
  "scan_coverage": { "start_mhz": 20, "end_mhz": 8000 },
  "entries": [
    { "freq_mhz": 97.4,  "bw_khz": 200, "max_dbm": -41.1 },
    { "freq_mhz": 433.92, "bw_khz": 25, "max_dbm": -68.3 }
  ]
}
```

### 7.2 任务指令格式（Cloud → Edge，WebSocket）

```json
{
  "task_id": "TASK-000123",
  "type": "scan",
  "params": {
    "center_mhz": 433.92,
    "span_mhz": 10,
    "analysis_bw_khz": 25,
    "duration_sec": 300
  },
  "stream": {
    "enabled": false,
    "fps": 0
  }
}
```

`stream.enabled = true` 时，Edge 同时推送实时频谱流（帧率受 `fps` 控制，0 = 最快）。

### 7.3 实时频谱帧格式（Edge → Cloud，WebSocket 二进制）

| 字段 | 类型 | 说明 |
|------|------|------|
| station_id | string | 站点 ID |
| task_id | string | 关联任务 ID |
| timestamp | int64 | Unix 时间戳（毫秒） |
| freq_start_hz | float64 | 起始频率 |
| freq_step_hz | float32 | 步进 |
| levels | float32[] | 电平数组（dBm） |

## 8. 已决策的设计要点

| 决策项 | 结论 | 原因 |
|--------|------|------|
| 连接方向 | Edge 主动连接 Cloud | 多站点部署方便，无需逐一配置 Cloud 的站点 IP |
| 设备驱动 | 抽象层隔离 | 兼容多厂商设备，切换设备只改配置 |
| 预处理位置 | 边缘做合并 + 聚合 | 减少上传带宽，Cloud 收到即可用 |
| 帧率控制 | 边缘节流 | 设备速度不可控，由软件层控制推送频率 |
| AI 策略 | 分层处理（模板→异常→AI→人工） | 直接端到端识别不可靠，分层保证可控性 |
| 快照存储 | 低频常规 + 触发高频 | 平衡存储成本与回放需求 |
| 时序数据库 | TimescaleDB | 原生时序查询，SQL 接口，降采样内置 |
| 信号库数据库 | PostgreSQL | 关系型，JSONB 灵活扩展 |
