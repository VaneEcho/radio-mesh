# RF·MESH 无线电监测网格系统 — 需求与设计说明

> 文档版本：v0.5 | 最后更新：2026-03-13
> 性质：研发对齐文档，记录架构决策与功能需求，随沟通持续更新

---

## 1. 系统定位

RF·MESH 是一套部署于专网环境的多站点无线电监测平台，由边缘采集节点与中心云端服务器协同构成。系统的核心目标是：

- **持续积累**：各站点不间断扫描，形成全频段历史电平数据库
- **异常发现**：通过模板匹配与 AI 辅助，自动识别频谱中的异常信号
- **任务驱动**：支持对特定频段发起定向扫描任务，聚合多站结果
- **台站建库**：逐步建立覆盖辖区内所有已知信号的台站信号库

## 2. 整体架构

### 2.1 拓扑结构

```
[ 监测设备（各厂商） ]
        │ SCPI / 私有协议
[ 边缘节点软件 (Edge Agent) ]    ← 每个监测站点部署一套
        │ 向上主动建立连接（WebSocket + REST）
        ↓
[ 云端服务器 (Cloud Server) ]    ← 专网内中心服务器
        │ WebSocket
[ 云端前端 (Web UI) ]
```

**连接方向约定：** Edge 主动向 Cloud 注册并建立长连接，Cloud 通过已建立的连接向 Edge 下发任务。这一设计简化了多站点部署，避免 Cloud 需要逐一主动连接各站点内网 IP。

**网络规划：** 通过隧道技术将各站点与云端服务器纳入同一扁平化局域网（不同网段），降低穿透复杂度。

### 2.2 设备驱动抽象层

系统需兼容多厂商监测设备，因此边缘软件设计**驱动抽象层**：

- 定义统一的设备驱动接口（如 `scan_range(start, stop, rbw)` → 返回标准格式频谱帧）
- 后续新设备仅需实现同一接口，通过配置文件切换驱动，不影响上层逻辑
- 各设备的特有限制（如单次扫描最大跨距、IF 分析带宽）由驱动层内部处理，上层无感

**已确认的适配设备：**

| 设备 | 接口 | 驱动方式 | 状态 |
|------|------|----------|------|
| R&S EM550 VHF/UHF 数字宽带接收机 | SCPI over Ethernet（TCP） | `RsInstrument` Python 库 + 自研 SCPI 命令封装 | **第一优先，已在站点部署** |
| Tektronix RSA306B USB 频谱分析仪 | USB 3.0 | `tekrsa-api-wrap`（Python，调用 `libRSA_API.so`） | 第二优先，设备已有，文档已找到 |

**R&S EM550 技术说明：**
- 覆盖范围：20 MHz – 3.6 GHz（VHF/UHF 监测接收机）
- 单次 IF 分析带宽：10 kHz – 9.6 MHz（**驱动层须将全频段任务分解为 ≤9.6 MHz 的切片，循环步进拼接**）
- 批量扫描模式（内置）：支持全频段自动步进，可通过 SCPI 配置扫描范围、步进和驻留时间
- 远程控制接口：SCPI over TCP/IP（标准以太网），端口通常为 `5025`
- Python 驱动库：[`RsInstrument`](https://github.com/Rohde-Schwarz/RsInstrument)（R&S 官方 Python 封装，无需安装 VISA，支持纯 Socket 通信）
- 文档获取：需联系 R&S 销售获取 EM550 编程手册（Programmer's Manual），可能附带于设备随机文档或通过 R&S 客户门户下载
- 软件兼容性：与 R&S EB200/ESMB 家族软件兼容，已有 Demo 可复用

**远程控制技术选型（通用 R&S SCPI 设备）：**

| 方案 | 工具 | 适用场景 |
|------|------|----------|
| 方案 A（推荐） | `RsInstrument` Python 库 + 原始 SCPI 字符串 | 适合 EM550 这类无独立 Python 驱动包的旧设备 |
| 方案 B | 设备专用 Python 驱动（如 `RsFsw`） | 适合 FSW 等有专属驱动的新设备，有命令自动补全 |
| 方案 C | 纯 Socket + 手写 SCPI 字符串 | 最轻量，无外部依赖，适合资源受限的边缘设备 |

**本项目选择方案 A（`RsInstrument`）** 作为 R&S 设备的统一底层，原因：
- R&S 官方维护，兼容所有 SCPI 设备
- 内置 OPC 同步（等待仪器完成测量后再读取，避免时序错误）
- 内置错误检查、重试机制、日志记录
- 无需安装 VISA 驱动（纯 Socket 模式），适合 Linux 边缘节点

**RSA306B 技术说明：**
- 覆盖范围：9 kHz – 6.2 GHz，最大实时捕获带宽 40 MHz
- Linux 下通过 Tektronix 官方 `libRSA_API.so` + NTIA 开源 Python 封装（`tekrsa-api-wrap`）驱动
- 参考资料：[RSA306B Programmer Manual (PDF)](https://download.tek.com/manual/RSA300-500-600-Series-Spectrum-Analyzers-API-Programming-Reference-077103104.pdf)、[tekrsa-api-wrap (GitHub)](https://github.com/NTIA/tekrsa-api-wrap)、[Tektronix 官方示例 (GitHub)](https://github.com/tektronix/RSA_API)
- 全频段宽扫时，驱动层循环调整中心频率，每次采集 40 MHz 切片并自动拼接

## 3. 边缘节点软件（Edge Agent）

### 3.1 常规全频段扫描

边缘节点在无任务下发时，持续执行全频段（20 MHz – 8 GHz）背景扫描。

**扫描参数：**
- 原始步进：25 kHz（全频段共约 319,200 个数据点）
- 驱动层按设备能力自动分段，上层统一请求全频段

**预处理（频段合并）：**
原始 25 kHz 步进不直接上传，按频段业务规划合并为信道粒度数据。合并规则以配置文件定义，可热更新。典型规则：

| 频段 | 业务 | 合并粒度 |
|------|------|----------|
| 87.5–108 MHz | FM 广播 | 200 kHz（8×25K） |
| 118–137 MHz | 航空 VHF | 25 kHz（保持原始精度） |
| 350–470 MHz | 对讲/PDT | 12.5 kHz / 25 kHz |
| 运营商 4G/5G 频段 | 移动通信 | 整段聚合（不按信道） |
| 其余兜底 | — | 100 kHz（4×25K） |

**1 分钟聚合上传：**
- 时间窗口：1 分钟
- 聚合方式：每个信道取窗口内最大电平值
- 上传内容：结构化聚合包（见 §7.1），体积远小于原始数据

### 3.2 任务执行模式

Cloud 下发扫描任务时，Edge 暂停常规扫描，执行任务扫描，完成后恢复常规扫描。任务参数包括：中心频率、跨距、分析带宽、扫描时长。

### 3.3 实时频谱流

当 Cloud 下发实时流任务时，Edge 对指定频段做连续扫描，并将频谱帧实时推送至 Cloud。

**帧率控制：**
- 设备本身以固有速度产出数据，Edge 软件在发送前进行节流
- 可配置目标帧率：
  - `0`：不节流，按设备最快速度发送
  - 默认值：5 fps
  - 推荐范围：5–10 fps（可在前端或任务参数中调整）
- 后台统计任务（非实时查看）：1 帧/分钟

## 4. 云端功能模块

### 4.1 站点总览仪表盘

- 展示所有已注册站点的在线状态、最后心跳时间、今日数据点数等
- 实时推送站点状态变更

### 4.2 频率指配工具

**场景：** 需要为某业务分配工作频率，查询现有频段内哪些信道当前空闲。

**输入参数：**
- 上行频段范围（起止频率）
- 下行频段范围（起止频率）
- 信道带宽
- 查询时间窗（最近 N 分钟历史数据）
- 空闲判定阈值（默认 -95 dBm，可调）
- 参与统计的站点（默认全选）

**输出：**
- 信道对照表：上下行对称展示，每个信道显示最大电平值及空闲/占用判断
- 可导出 CSV，供出具频率分配报告使用

### 4.3 批量站点扫描任务

**场景：** 对特定频段发起面向全部或指定站点的定向扫描，汇聚多站结果，判断信号的空间分布。

**输入参数：**
- 中心频率
- 跨距（扫描范围 = 中心频率 ± 跨距/2）
- 分析带宽（≤ 跨距，统计此带宽内的最大值）
- 扫描时长
- 参与站点（默认全选）

**执行流程：**
1. Cloud 将任务下发至各选中站点
2. 各 Edge 执行扫描，统计分析带宽内最大电平值，完成后上报结果
3. Cloud 汇聚展示：站点列表 + 各站最大电平对比

**实时回放：**
在汇总结果页，点击任意站点的「实时回放」按钮，触发该站点启动实时频谱流，前端显示滚动频谱图/瀑布图。回放参数沿用任务的频段设置，帧率可单独调整。

### 4.4 历史数据回放

**场景：** 回溯某频段在过去某段时间内的信号变化情况。

**输入参数：**
- 中心频率
- 分析带宽（可另加可选跨距，用于查看历史频谱图）
- 历史时间范围（起止时间）
- 优先展示站点（默认展示全部）

**展示层 1 — 电平时间线（折线图）：**
- 横轴：所选时间范围
- 纵轴：最大电平（dBm）
- 每个站点一条折线（可切换单站/多站叠加）
- 鼠标悬停显示精确时间与电平值
- 可快速识别信号何时出现、何时消失、强弱变化规律

**展示层 2 — 历史频谱快照（点击展开）：**
- 点击时间线上任意时刻 → 弹出该时刻的频谱图
- 展示以中心频率为中心、跨距范围内的完整频谱形态
- 用于判断信号特征：宽带/窄带、持续发射/跳频突发、单一/多源等
- 要求边缘在常规扫描时以较低频率存储全频谱快照（或至少存储触发条件下的快照）

## 5. AI 辅助信号分析

### 5.1 AI 模型选型策略

AI 图像识别采用**可插拔多后端设计**，通过统一接口支持不同阶段切换模型：

| 阶段 | 模型 | 说明 |
|------|------|------|
| 初期调试 | Qwen2.5-VL 3B（本地，欧拉 OS 部署） | 支持图像输入，效果有限，仅用于联调接口 |
| 生产环境 | OpenAI GPT-4o / Gemini / Kimi / 豆包（可切换） | 多模态效果好，通过配置文件指定后端 |

接口层统一输入格式（频谱图 + 瀑布图 + 元数据 JSON），与具体模型解耦，切换模型只改配置。

### 5.2 设计思路

直接用 AI 对全频段频谱图做端到端的信号识别（数量、位置、带宽）技术难度极高，工程上不可靠。采用**分层处理**方式：

```
原始频谱数据
    │
    ▼
[第一层：模板匹配初筛]
    按频段规划模板，逐信道判断是否有信号占用
    → 输出：信号列表（频率、估算带宽）
    │
    ▼
[第二层：异常检测]
    对比信道规划：
      - 带宽异常（如 25K 对讲频段出现 100K 宽信号）
      - 电平异常（超出历史基线）
      - 频率偏移异常（信号不在标准信道中心）
    → 输出：异常信号列表，提取频谱截图 + 瀑布图截图
    │
    ▼
[第三层：AI 初判]
    将异常信号的频谱图 + 瀑布图送入图像识别 AI
    结合频段规划知识库（可联网检索补充）
    → 输出：信号类型初步判断 + 置信度 + 依据说明
    │
    ▼
[第四层：人工核实]
    技术人员对 AI 初判结果进行核实，确认或修正
    → 确认后入库，更新台站信号库
```

### 5.3 台站信号库建设目标

台站信号库是系统长期积累的核心资产，分三类管理：

| 类别 | 说明 | 处理方式 |
|------|------|----------|
| AI 自动归档 | 置信度高、特征明确的已知信号 | AI 直接入库，人工抽查 |
| AI + 人工核实 | AI 有初判但置信度中等 | 推送待核实队列，人工确认后入库 |
| 人工处理 | 高异常程度或 AI 无法判断 | 推送处置工单，人工分析后手动录入 |

最终目标：辖区内每个出现过的信号都有对应记录，包含频率、带宽、调制方式、业务归属、历史出现规律等。

## 6. 数据存储设计

### 6.1 存储层次划分

**存储总原则：** 不在软件层设置截断，存储容量由硬件扩容解决，软件只管写入与查询效率。

| 数据类型 | 存储方式 | 保留策略 | 说明 |
|----------|----------|----------|------|
| 1 分钟聚合统计包 | TimescaleDB（时序主表） | **最少 3 个月**，不自动删除 | 用于历史电平趋势查询、频率指配 |
| 全频谱快照 | 文件系统（压缩，按日期分目录） | **最少 3 个月**，不自动删除 | 用于历史回放点击查看频谱图 |
| 触发快照（告警/任务） | 同上，独立子目录 | 永久保留 | 证据留存，与告警/任务记录关联 |
| 台站信号库 | PostgreSQL（关系型） | 永久 | 核心业务数据 |
| 任务记录 | PostgreSQL | 永久 | 可追溯 |
| 告警记录 | PostgreSQL | 永久 | 可追溯 |

### 6.2 核心数据模型

**时序聚合表（spectrum_stats）：**
```
time          TIMESTAMPTZ   -- 分钟级时间戳
station_id    TEXT
freq_mhz      FLOAT         -- 信道中心频率
bw_khz        FLOAT         -- 信道带宽
max_dbm       FLOAT         -- 该分钟内最大电平
```

**全频谱快照（spectrum_snapshots）：**
```
snapshot_id   UUID
station_id    TEXT
captured_at   TIMESTAMPTZ
freq_start    FLOAT         -- 快照频率起点（MHz）
freq_end      FLOAT         -- 快照频率终点（MHz）
resolution    FLOAT         -- 实际分辨率（kHz）
data_path     TEXT          -- 压缩数据文件路径
```

**台站信号库（signals）：**
```
signal_id         TEXT (主键, SIG-XXXXXX)
freq_center_mhz   FLOAT
bandwidth_khz     FLOAT
modulation        TEXT
signal_type       TEXT         -- 业务类型
status            ENUM(auto_archived, pending_review, manual)
confidence        FLOAT        -- AI 置信度
first_seen        TIMESTAMPTZ
last_seen         TIMESTAMPTZ
occurrence_count  INT
ai_conclusion     TEXT
tags              TEXT[]
notes             TEXT
```

### 6.3 全频谱快照的存储策略

全频段快照数据量大（每站每帧约 1–2 MB 压缩后），不宜对每帧都持久化。建议：

- **常规模式**：以低频率（如每 5 分钟一次）存储一帧全频谱快照，用于历史回放点击查看
- **任务/告警触发模式**：任务执行期间或异常触发时，提高快照存储频率（如每分钟或每次扫描完成即存）
- **存储格式**：频率轴 + 电平数组，使用 NumPy `.npz` 或 MessagePack 压缩，避免 JSON 冗余

## 7. 关键接口协议

### 7.1 聚合包格式（Edge → Cloud，每分钟上传）

```json
{
  "station_id": "ST-01",
  "timestamp": "2026-03-13T08:00:00Z",
  "duration_sec": 60,
  "scan_coverage": { "start_mhz": 20, "end_mhz": 8000 },
  "entries": [
    { "freq_mhz": 97.4,  "bw_khz": 200, "max_dbm": -41.1 },
    { "freq_mhz": 433.92, "bw_khz": 25, "max_dbm": -68.3 }
  ]
}
```

### 7.2 任务指令格式（Cloud → Edge，WebSocket）

```json
{
  "task_id": "TASK-000123",
  "type": "scan",
  "params": {
    "center_mhz": 433.92,
    "span_mhz": 10,
    "analysis_bw_khz": 25,
    "duration_sec": 300
  },
  "stream": {
    "enabled": false,
    "fps": 0
  }
}
```

`stream.enabled = true` 时，Edge 同时推送实时频谱流（帧率受 `fps` 控制，0 = 最快）。

### 7.3 实时频谱帧格式（Edge → Cloud，WebSocket 二进制）

| 字段 | 类型 | 说明 |
|------|------|------|
| station_id | string | 站点 ID |
| task_id | string | 关联任务 ID |
| timestamp | int64 | Unix 时间戳（毫秒） |
| freq_start_hz | float64 | 起始频率 |
| freq_step_hz | float32 | 步进 |
| levels | float32[] | 电平数组（dBm） |

## 8. 已决策的设计要点

| 决策项 | 结论 | 原因 |
|--------|------|------|
| 连接方向 | Edge 主动连接 Cloud | 多站点部署方便，无需逐一配置 Cloud 的站点 IP |
| 设备驱动 | 抽象层隔离 | 兼容多厂商设备，切换设备只改配置 |
| 预处理位置 | 边缘做合并 + 聚合 | 减少上传带宽，Cloud 收到即可用 |
| 帧率控制 | 边缘节流 | 设备速度不可控，由软件层控制推送频率 |
| AI 策略 | 分层处理（模板→异常→AI→人工） | 直接端到端识别不可靠，分层保证可控性 |
| 快照存储 | 低频常规 + 触发高频 | 平衡写入 IO 与回放需求 |
| 数据保留 | 不设软件截断，最少 3 个月 | 容量由硬件扩容保障，软件不限制 |
| 时序数据库 | TimescaleDB | 原生时序查询，SQL 接口，降采样内置 |
| 信号库数据库 | PostgreSQL | 关系型，JSONB 灵活扩展 |
| AI 图像识别 | 可插拔多后端（本地 Qwen → OpenAI/Gemini/Kimi） | 调试用本地模型，生产切换云端多模态 |
| 前端框架 | Vue 3 + Vite + Element Plus + ECharts | 中文生态成熟，ECharts 图表能力强 |
| R&S 设备接口 | `RsInstrument` + 原始 SCPI 命令 | 官方维护，纯 Socket 无需 VISA，适合 Linux |
| 频段规则管理 | 后台可视化配置页面 + `band_rules.yaml`，不写死于代码 | 频率划分调整时只需在后台修改，无需改代码 |

## 9. 频段规则管理后台

### 9.1 设计背景

无线电频率划分规定随 ITU 世界无线电通信大会（约每 3–4 年修订一次）及国内主管部门决定动态调整，频段合并规则不能硬编码于程序逻辑中。

### 9.2 频段规则数据结构

每条频段规则包含以下字段：

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| `id` | 自增整数 | 规则 ID | 42 |
| `freq_start_mhz` | 浮点 | 频段起始频率（MHz） | 87.5 |
| `freq_end_mhz` | 浮点 | 频段结束频率（MHz） | 108.0 |
| `service_name` | 字符串 | 业务名称 | FM广播 |
| `service_type` | 枚举 | 业务大类 | broadcasting |
| `channel_bw_khz` | 浮点 | 聚合粒度（kHz），即合并步进 | 200 |
| `authority` | 枚举 | 管理权限层级 | national / provincial / local |
| `operator` | 字符串（可空） | 归属运营商（移动通信专用） | 中国移动 |
| `description` | 字符串 | 备注说明 | — |
| `anomaly_bw_threshold_khz` | 浮点（可空） | 信号宽度超出此值时触发异常告警 | 500 |
| `enabled` | 布尔 | 是否启用此规则 | true |
| `updated_at` | 时间戳 | 最后修改时间 | — |

`service_type` 枚举值：`broadcasting / aviation / maritime / pmr / mobile / gnss / satellite / amateur / radar / ism / government / other`

### 9.3 管理后台页面功能

**频段规则列表页：**
- 按频率范围排序展示所有规则
- 支持按 `service_type`、`authority`、关键词筛选
- 每行显示：频率范围、业务名称、信道带宽、权限层级、启用状态、最后修改时间
- 操作：编辑 / 禁用 / 删除

**频段规则编辑页：**
- 表单填写上述字段
- 频率范围自动校验（不得与已有规则重叠）
- 修改即时生效（Edge 下次扫描时拉取最新规则，或通过 WebSocket 推送规则更新）

**频谱可视化预览（可选，后期添加）：**
- 横轴为 20 MHz – 8 GHz 的频率轴，纵向展示各频段颜色块
- 悬停显示业务名称和信道带宽
- 直观检查规则是否有遗漏或重叠

### 9.4 规则配置文件（`band_rules.yaml`）

后台数据库以 YAML 文件形式导出/导入，便于版本管理和跨环境同步。初始版本见 `docs/band_rules.yaml`，内容基于《中华人民共和国无线电频率划分规定》（工信部令第 62 号，2023 年 7 月 1 日施行）及各运营商公开频段资料整理，未来通过后台页面动态维护。
